# Plan 02-01: Guardrails Engine + EIP-7702 Session Keys + CRUD API

## Objective

Implement configurable safety rules enforced on every transaction, EIP-7702 session key verification, PostgreSQL repository layer, and CRUD API endpoints for agents, guardrail rules, and session keys.

## Tasks

### Task 1: PostgreSQL repository layer
- Add sqlx dependency (runtime queries, not compile-time macros)
- Create migration: agents, session_keys, guardrail_rules, spend_ledger tables
- Implement repo modules: agents, guardrails, session_keys, spend_ledger
- DB CHECK constraints: max_spend > 0, spent >= 0, spent <= max_spend, amount > 0

### Task 2: Domain models
- Agent model (id, owner_address, name, is_active, created_at)
- SessionKey model (id, agent_id, public_key, max_spend, spent, allowed_contracts, expires_at, is_revoked)
- GuardrailRule model with RuleType enum: MaxSpendPerTx, MaxSpendPerDay, AllowedContracts, MaxLeverage, MaxSlippage

### Task 3: Guardrails engine middleware
- `evaluate()` function: takes rules + transaction context, returns pass/fail
- Fail-closed: any rule failure blocks the transaction
- 5 rule types with parameter extraction from JSON
- Inactive rules skipped, multiple rules evaluated sequentially

### Task 4: EIP-7702 session key verification
- Verify session key not revoked
- Verify not expired
- Verify spend within limits (max_spend - spent >= payment amount)
- Verify target contract in allowed_contracts whitelist

### Task 5: CRUD API endpoints
- POST/GET /api/v1/agents — create and list agents
- POST/GET /api/v1/agents/{id}/rules — create and list guardrail rules
- PUT/DELETE /api/v1/agents/{id}/rules/{rule_id} — update and deactivate rules
- POST/GET /api/v1/agents/{id}/session-keys — create and list session keys
- GET/DELETE /api/v1/agents/{id}/session-keys/{key_id} — get and revoke keys

### Task 6: Proxy handler integration
- Wire guardrails evaluation into proxy handler
- Wire session key verification into proxy handler
- M4: Resource URL validation (scheme + host + port + path segment boundary)
- M5: Asset address validation against canonical USDC per network
- Reserve-then-forward spend pattern (record spend BEFORE forwarding)

### Task 7: Security hardening (from Level-10 audit)
- C1-C3: Atomic reserve-then-forward (fail-closed spend recording)
- C4: Reject payment amounts exceeding u64
- C5: Checked i64/u64 casts in all repo functions
- H2: Cross-agent authorization on update/delete/revoke
- H4: URL segment-boundary matching
- M3: Checked u32 casts for leverage/slippage
- M6: Inactive agent check before proxy forwarding

## Success Criteria
- MaxSpendPerTx blocks oversized transactions
- MaxSpendPerDay enforces rolling 24h window
- AllowedContracts blocks unauthorized targets
- Session keys verified: spend limits, contract whitelist, expiry, revocation
- Cross-agent access prevented on all CRUD operations
- 52 total tests pass (19 Phase 1 + 33 Phase 2)
