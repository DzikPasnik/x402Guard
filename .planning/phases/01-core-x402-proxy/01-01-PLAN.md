# Plan 01-01: x402 Types and EIP-3009 Verification

## Objective

Implement the core x402 protocol types and EIP-3009 signature verification in Rust. This is the cryptographic heart of the proxy.

## Tasks

### Task 1: Add dependencies to Cargo.toml
- Add `base64` crate (for base64url header encoding)
- Add `hex` crate (for hex address/nonce parsing)
- Verify `alloy` features include `sol-types` for EIP-712

### Task 2: Create x402 types module
- Create `proxy/src/middleware/x402/` directory (convert x402.rs to directory module)
- `types.rs`: Define Rust structs matching the x402 protocol:
  - `PaymentRequirements` { scheme, network, max_amount_required, resource, description, pay_to, max_timeout_seconds, asset, extra }
  - `PaymentPayload` { scheme, network, payload: PaymentProof }
  - `PaymentProof` { signature, authorization: TransferAuthorization }
  - `TransferAuthorization` { from, to, value, valid_after, valid_before, nonce }
- Use serde for JSON deserialization with camelCase rename
- Parse amounts as String (USDC uses 6 decimal U256 representation)

### Task 3: Implement EIP-712 / EIP-3009 signature verification
- `verify.rs`: Implement `verify_transfer_authorization()`
  - Build the EIP-712 domain separator for USDC on Base (name: "USD Coin", version: "2", chainId: 8453/84532)
  - Build the TransferWithAuthorization type hash
  - Use `alloy::sol!` macro to define the EIP-712 struct
  - Recover signer address from signature using `alloy::primitives::Signature`
  - Compare recovered address against `authorization.from`
  - Verify temporal bounds: `valid_after <= now <= valid_before`
  - Verify `value >= max_amount_required`
  - Verify `to == pay_to`
  - Return `Result<VerifiedPayment, AppError>`

### Task 4: Base64url header parsing
- `mod.rs`: Implement `parse_payment_requirements(header: &str) -> Result<PaymentRequirements>`
- `mod.rs`: Implement `parse_payment_payload(header: &str) -> Result<PaymentPayload>`
- Base64url decode, then JSON deserialize

### Task 5: Unit tests
- Test valid EIP-3009 signature verification (use a known test vector)
- Test invalid signature rejection
- Test expired authorization rejection (valid_before in past)
- Test amount mismatch rejection
- Test base64url parsing of both headers
- Test malformed JSON rejection

## Success Criteria
- `PaymentRequirements` and `PaymentPayload` serialize/deserialize correctly
- EIP-712 signature recovery matches known test vectors
- All 6+ unit tests pass
