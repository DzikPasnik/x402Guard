# Plan 01-02: Rate Limiting and Nonce Deduplication

## Objective

Implement Redis-backed sliding window rate limiting and nonce deduplication to prevent replay attacks and DDoS.

## Tasks

### Task 1: Implement sliding window rate limiter
- In `rate_limit.rs`, implement `RateLimiter` struct wrapping Redis `MultiplexedConnection`
- Method: `check_rate_limit(key: &str, max_requests: u32, window_secs: u64) -> Result<RateLimitResult>`
- Algorithm: Sorted set sliding window
  - `ZADD key timestamp timestamp` — add request
  - `ZREMRANGEBYSCORE key 0 (now - window)` — evict old
  - `ZCARD key` — count in window
- Return `RateLimitResult { allowed: bool, remaining: u32, retry_after_secs: Option<u64> }`
- Support both global (by proxy IP) and per-agent (by agent ID) keys

### Task 2: Create Axum middleware layer for rate limiting
- Implement `RateLimitLayer` as a Tower layer
- Extract agent identifier from request (header `X-Agent-Id` or session key)
- Check global rate limit first, then per-agent
- On exceed: return 429 with `Retry-After` header and JSON error body
- On pass: call next handler

### Task 3: Implement nonce deduplication
- Create `nonce.rs` module
- Method: `check_and_store_nonce(nonce: &str, ttl_secs: u64) -> Result<bool>`
- Use `SET key 1 NX EX ttl_secs` — returns true if nonce was new, false if replay
- Key format: `nonce:{hex_nonce}`
- TTL should match `maxTimeoutSeconds` from payment requirements (default 300s)

### Task 4: Unit and integration tests
- Test rate limiter allows requests under limit
- Test rate limiter blocks at limit, returns correct Retry-After
- Test window slides (old requests expire)
- Test nonce deduplication blocks same nonce
- Test nonce allows different nonces
- Test nonce expires after TTL
- Mock Redis or use test container pattern

## Success Criteria
- Rate limiting blocks excess requests with 429 + Retry-After
- Nonce dedup prevents same nonce from being used twice
- Per-agent and global limits work independently
