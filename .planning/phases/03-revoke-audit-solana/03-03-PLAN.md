# Plan 03-03: Solana Anchor Guard Program

## Goal

Create a standalone Anchor program implementing a PDA vault with on-chain guardrails for Solana USDC. The program enforces per-tx limits, daily caps, program whitelists, and agent revocation — mirroring the EVM guardrails on-chain.

## Requirements Covered

- FR-4.1: Initialize PDA vault per user with configurable rules
- FR-4.2: Deposit USDC into PDA vault
- FR-4.3: Guarded withdrawal with per-tx limit, daily cap, program whitelist
- FR-4.4: Revoke agent access (zero out agent pubkey)
- FR-4.5: Update vault rules (owner-only)
- FR-4.6: Deploy guard program on Solana devnet

## New Dependencies (Separate Workspace)

The Anchor program lives in `solana/` — a completely separate Cargo workspace from `proxy/`. It does NOT share dependencies with the proxy.

- `anchor-lang = "0.30"` (VERIFY latest at implementation time)
- `anchor-spl = "0.30"` (VERIFY, must match anchor-lang)

## Critical Constraints

1. **Separate workspace:** `solana/` has its own `Cargo.toml` workspace. NOT added to root workspace members (BPF target conflicts).
2. **Separate build:** `anchor build` (not `cargo test`). Requires Anchor CLI + Solana CLI.
3. **serde pin:** The proxy pins serde `=1.0.219`. The Anchor workspace uses whatever anchor-lang requires. No conflict because separate workspaces.
4. **USDC mint:** Configurable — use custom mint for devnet testing, verify mainnet address.

## Implementation Steps

### Step 1: Scaffold Anchor Project

Create `solana/` directory with:
- `Anchor.toml` — cluster config (localnet for tests, devnet for deploy)
- `Cargo.toml` — workspace with `programs/x402-guard`
- `programs/x402-guard/Cargo.toml` — anchor-lang, anchor-spl
- `package.json` — @coral-xyz/anchor, @solana/web3.js, mocha, chai, ts-mocha
- `tsconfig.json` — standard Anchor test config

### Step 2: VaultState Account + Constants + Errors

**New files:**
- `solana/programs/x402-guard/src/state/vault.rs`
- `solana/programs/x402-guard/src/constants.rs`
- `solana/programs/x402-guard/src/errors.rs`

VaultState fields:
- `owner: Pubkey` — vault owner (can update rules, revoke, close)
- `agent: Pubkey` — authorized agent (Pubkey::default() = revoked)
- `max_spend_per_tx: u64` — per-transaction USDC limit (6 decimals)
- `max_spend_per_day: u64` — daily USDC cap
- `spent_today: u64` — amount spent in current window
- `day_window_start: i64` — unix timestamp of window start
- `agent_expires_at: i64` — agent key expiration (0 = no expiry)
- `allowed_programs: Vec<Pubkey>` — whitelisted destinations (max 10)
- `is_active: bool` — pause toggle
- `bump: u8` — PDA bump seed
- `_reserved: [u8; 64]` — future fields without realloc

SPACE calculation: 8 + 32 + 32 + 8 + 8 + 8 + 8 + 8 + (4 + 32*10) + 1 + 1 + 64 = 502 bytes

GuardError variants:
- Unauthorized, VaultPaused, AgentExpired, ZeroAmount
- ExceedsPerTxLimit, ExceedsDailyCap, ProgramNotWhitelisted
- InvalidAgentPubkey, InvalidLimits, TooManyPrograms, ArithmeticOverflow

### Step 3: initialize_vault Instruction

**New file:** `solana/programs/x402-guard/src/instructions/initialize_vault.rs`

Accounts:
- `owner: Signer` (mutable, payer)
- `vault: Account<VaultState>` (init, PDA seeds=["vault", owner], space=SPACE)
- `vault_token_account: Account<TokenAccount>` (init, ATA for vault+usdc_mint)
- `usdc_mint: Account<Mint>`
- `token_program, associated_token_program, system_program`

Validation:
- `agent != Pubkey::default()`
- `max_spend_per_tx > 0`, `max_spend_per_day > 0`
- `max_spend_per_day >= max_spend_per_tx`
- `allowed_programs.len() <= MAX_ALLOWED_PROGRAMS`

### Step 4: deposit Instruction

**New file:** `solana/programs/x402-guard/src/instructions/deposit.rs`

Simple SPL Token transfer from user's ATA to vault's ATA. Any user can deposit (not restricted to owner).

Accounts:
- `depositor: Signer` (mutable)
- `vault: Account<VaultState>` (seeds validation only, not mutable)
- `depositor_token_account: Account<TokenAccount>` (mutable)
- `vault_token_account: Account<TokenAccount>` (mutable, ATA for vault)
- `usdc_mint: Account<Mint>`
- `token_program`

### Step 5: guarded_withdraw Instruction

**New file:** `solana/programs/x402-guard/src/instructions/guarded_withdraw.rs`

This is the core instruction — all guardrails enforced here.

Accounts:
- `agent: Signer` — must match `vault.agent`
- `vault: Account<VaultState>` (mutable, PDA seeds+bump)
- `vault_token_account: Account<TokenAccount>` (mutable, ATA for vault)
- `destination_token_account: Account<TokenAccount>` (mutable)
- `usdc_mint: Account<Mint>`
- `token_program`

Checks (all must pass — fail-closed):
1. `vault.is_active == true`
2. `agent.key() == vault.agent`
3. Agent not expired (if `agent_expires_at > 0`, check `clock < expires_at`)
4. `amount > 0`
5. `amount <= vault.max_spend_per_tx`
6. Daily window reset (if 86400s elapsed, reset spent_today)
7. `spent_today + amount <= vault.max_spend_per_day` (checked_add)
8. Program whitelist: `allowed_programs.contains(&destination.owner)` (skip if empty)

Effects: `vault.spent_today = new_total` (update AFTER all checks pass)

Interactions:
- Execute `token::transfer` CPI from `vault_token_account` to `destination_token_account`
- Sign CPI with PDA signer seeds: `&[b"vault", owner_key.as_ref(), &[vault.bump]]`
- Emit optional `WithdrawExecuted` event with amount and destination for off-chain audit

### Step 6: update_rules Instruction

**New file:** `solana/programs/x402-guard/src/instructions/update_rules.rs`

Owner-only. Optional fields (only update provided values):
- `new_max_spend_per_tx: Option<u64>`
- `new_max_spend_per_day: Option<u64>`
- `new_allowed_programs: Option<Vec<Pubkey>>`
- `new_agent: Option<Pubkey>`
- `new_agent_expires_at: Option<i64>`
- `new_is_active: Option<bool>`

Validation: same constraints as initialize (>0, day >= tx, programs <= MAX).

### Step 7: revoke_agent Instruction

**New file:** `solana/programs/x402-guard/src/instructions/revoke_agent.rs`

Owner-only kill switch:
- `vault.agent = Pubkey::default()`
- `vault.is_active = false`
- `vault.agent_expires_at = 0`
- Emit `AgentRevoked` event

### Step 8: close_vault Instruction

**New file:** `solana/programs/x402-guard/src/instructions/close_vault.rs`

Owner-only. Closes vault and transfers remaining USDC + rent lamports to owner.

**Security note:** The program whitelist is NOT checked during close_vault. The owner can always recover their funds by closing the vault, regardless of `allowed_programs`. The whitelist only applies to `guarded_withdraw` by agents. This is intentional — the owner must never be locked out of their own funds.

Steps:
- Transfer all USDC from vault ATA to owner ATA via CPI (PDA signer)
- Close vault ATA (transfer remaining lamports to owner)
- Close vault PDA account (Anchor `close = owner` zeros discriminator, prevents resurrection)

### Step 9: lib.rs Program Entrypoint

**New file:** `solana/programs/x402-guard/src/lib.rs`

`#[program]` module dispatching to instruction handlers:
- `initialize_vault`
- `deposit`
- `guarded_withdraw`
- `update_rules`
- `revoke_agent`
- `close_vault`

### Step 10: TypeScript Integration Tests

**New file:** `solana/tests/x402-guard.ts`

Tests (run via `anchor test`):
1. Initialize vault with valid params → success
2. Deposit USDC → vault balance increases
3. Guarded withdraw within limits → success, spent_today updated
4. Guarded withdraw exceeding per-tx limit → error
5. Guarded withdraw exceeding daily cap → error
6. Guarded withdraw to non-whitelisted destination → error
7. Guarded withdraw with expired agent → error
8. Guarded withdraw after revoke → error (agent = default)
9. Update rules (owner) → success
10. Update rules (non-owner) → error
11. Revoke agent → agent zeroed, vault paused
12. Close vault → USDC returned, accounts closed
13. Daily window reset after 24h → spent_today resets
