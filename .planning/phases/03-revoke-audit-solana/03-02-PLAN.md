# Plan 03-02: Revoke System for Base (EVM)

## Goal

Implement one-click revoke-all for session keys: instant DB revocation + agent deactivation + unsigned EIP-7702 authorization data for client-side on-chain revocation. Non-custodial — proxy never holds EOA keys.

## Requirements Covered

- FR-3.4: Revoke individual session keys (enhanced with audit logging)
- FR-3.5: Revoke ALL session keys for an agent (one-click, EIP-7702 zero-address delegation data)

## New Dependencies

None for Phase 3. On-chain submission deferred to Phase 4 (requires user wallet).

## Implementation Steps

### Step 1: Batch Revocation Repository Function

**Modified:** `proxy/src/repo/session_keys.rs`

Add:
```rust
pub async fn revoke_all_by_agent(pool: &PgPool, agent_id: Uuid) -> Result<u64>
```
- `UPDATE session_keys SET is_revoked = true WHERE agent_id = $1 AND is_revoked = false`
- Returns `rows_affected()` for audit logging

### Step 2: Agent Deactivation Repository Function

**Modified:** `proxy/src/repo/agents.rs`

Add:
```rust
pub async fn deactivate(pool: &PgPool, agent_id: Uuid) -> Result<()>
```
- `UPDATE agents SET is_active = false WHERE id = $1`
- Verify `rows_affected() > 0`

### Step 3: Revoke-All API Endpoint

**Modified:** `proxy/src/handlers/session_keys.rs`
**Modified:** `proxy/src/router.rs` — add route

Add handler:
```
POST /api/v1/agents/:agent_id/revoke-all
```

Response type `RevokeAllResponse`:
- `success: bool`
- `keys_revoked: u64`
- `agent_deactivated: bool`
- `on_chain_authorization: Option<serde_json::Value>` — unsigned EIP-7702 data

Flow:
1. Verify agent exists (404 if not)
2. **SECURITY [H2]:** Verify caller owns the agent — check `agent.owner_address` matches request context (401 if mismatch). Uses same cross-agent auth pattern as existing individual revoke handler.
3. Wrap Steps 4-5 in a single sqlx transaction for atomicity:
4. `revoke_all_by_agent(pool, agent_id)` — synchronous, must succeed
5. `deactivate(pool, agent_id)` — synchronous
6. Emit `AllSessionKeysRevoked` audit event (from 03-01) with `keys_revoked` count in metadata
7. Call `create_revoke_authorization_data(chain_id)` from services/revocation.rs and include in `RevokeAllResponse.on_chain_authorization`

### Step 4: Enhance Individual Revoke with Audit Logging

**Modified:** `proxy/src/handlers/session_keys.rs`

In existing `revoke_session_key` handler:
- After successful DB revoke, emit `SessionKeyRevoked` audit event
- Include `key_id`, `agent_id` in metadata

### Step 5: EIP-7702 Authorization Data Helper

**New file:** `proxy/src/services/revocation.rs`
**Modified:** `proxy/src/services/mod.rs` — add `pub mod revocation;`

Function:
```rust
pub fn create_revoke_authorization_data(chain_id: u64, eoa_nonce_hint: Option<u64>) -> serde_json::Value
```
- Returns unsigned EIP-7702 authorization JSON: `{ chain_id, address: "0x0..0", nonce: <hint_or_null>, message: "Sign to revoke all delegation..." }`
- `nonce` is the EOA's current nonce (fetched from RPC if available, otherwise null — dashboard fetches it client-side)
- Non-custodial: user/dashboard signs this with their EOA key
- Phase 4 dashboard will use this to trigger wallet signing via MetaMask/RainbowKit

### Step 6: Tests

Unit tests:
