# Plan 03-04: Proxy ↔ Solana Integration

## Goal

Connect the Rust proxy to the Solana guard program deployed in 03-03. The proxy queries vault state (read-only) via JSON-RPC, validates guardrails locally, and forwards x402 requests to the target service. The proxy is non-custodial: it never holds Solana keypairs or submits transactions. Agents sign and submit `guarded_withdraw` transactions independently to the Solana network.

## Requirements Covered

- FR-4.3 (proxy-side): Proxy routes Solana x402 payments through the guard program
- FR-4.6 (integration): End-to-end proxy → Solana devnet flow
- FR-6.1 (Solana events): Audit log captures Solana-side events

### Step 1b: Reqwest JSON-RPC Fallback (if Step 1 fails)

**New file:** `proxy/src/services/solana_rpc.rs`

If solana-sdk conflicts with `serde = "=1.0.219"`, use a lightweight JSON-RPC client via existing `reqwest`. Since proxy is read-only (non-custodial), only read operations are needed:

Functions:
- `get_account_info(http_client, rpc_url, pubkey_b58) → Result<Option<Vec<u8>>>`
  - POST `getAccountInfo` with base64 encoding, returns raw account bytes
  - Manually skip 8-byte Anchor discriminator, then borsh-deserialize VaultState
  - VaultState borsh schema is simple fixed-size fields (no nested structs) — hand-rolled deserialization is ~50 lines
- `get_token_account_balance(http_client, rpc_url, ata_b58) → Result<u64>`
  - POST `getTokenAccountBalance`, parse `amount` from JSON response

NOT needed for Phase 3 MVP (proxy doesn't submit transactions):
- ~~`send_transaction`~~ — agents submit `guarded_withdraw` directly
- ~~`get_latest_blockhash`~~ — only needed if proxy signs transactions

This fallback adds zero new dependencies and covers the read-only proxy scope.

### Step 3: Vault Status API Endpoint

**New file:** `proxy/src/handlers/solana_vault.rs`
**Modified:** `proxy/src/handlers/mod.rs` — add `pub mod solana_vault;`
**Modified:** `proxy/src/router.rs` — add routes

Endpoints:
```
GET /api/v1/solana/vault/:owner_pubkey
```

Response (VaultStatusResponse):
- `owner: String`
- `agent: String`
- `is_active: bool`
- `max_spend_per_tx: u64`
- `max_spend_per_day: u64`
- `spent_today: u64`
- `usdc_balance: u64`
- `agent_expires_at: i64`
- `allowed_programs: Vec<String>`

Implementation:
1. Derive vault PDA from owner pubkey + program ID
2. Fetch account data via RPC (`getAccountInfo`)
3. Deserialize VaultState (Anchor discriminator + borsh)
4. Fetch vault ATA balance
5. Return combined status

### Step 5: Proxy Solana Payment Flow (MVP — Read-Only)

**Modified:** `proxy/src/handlers/proxy.rs`

When proxy detects a Solana x402 payment (network = "solana-mainnet" or "solana-devnet"):
1. Extract `vault_owner` pubkey from x402 payment headers (new field: `x-402-solana-vault-owner`)
2. Query vault state via RPC to verify: vault exists, `is_active`, agent not expired
3. Validate amount against vault's `max_spend_per_tx` and remaining daily capacity (`max_spend_per_day - spent_today`)
4. Validate destination against vault's `allowed_programs` whitelist
5. Forward the payment request to the target service
6. Emit `SolanaWithdrawSubmitted` audit event

**Non-custodial scope:** The proxy performs read-only validation and request forwarding only. It does NOT submit the `guarded_withdraw` Solana transaction — the agent signs and submits that independently to Solana RPC. The on-chain guard program provides the ultimate enforcement. The proxy adds a pre-flight check layer (defense-in-depth).

**x402 Solana extension fields (MVP):**
- `x-402-network: "solana-devnet"` or `"solana-mainnet"` — identifies chain
- `x-402-solana-vault-owner: <base58 pubkey>` — vault owner for PDA derivation
- Standard x402 fields (`amount`, `target_url`, etc.) remain unchanged

| Risk | Mitigation |
|------|------------|
| solana-sdk conflicts with serde pin | Feature-gate + fallback to raw reqwest RPC |
| solana-sdk pulls in blst | Check cargo tree -i blst; use reqwest fallback |
